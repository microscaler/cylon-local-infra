# ms02

Non-Spark dev host. Runs Docker CE (buildx, compose), Kind, `act`, and remote IDE.

| Field | Value |
|---|---|
| LAN IP (wired `eno4`) | `192.168.1.189` |
| Ansible group | `dev_hosts` |
| Ansible user | `root` |
| Login user for tunnels / kind / Tilt | `casibbald` |
| Playbooks | `playbooks/dev_hosts.yml`, `playbooks/docker_dev_engine.yml` |
| Mac SSH aliases | `ms02` (casibbald), `ms02-root` (root), `ms02-dev-tunnel` (casibbald + forwards) — see `~/.ssh/config.d/ms02{,-dev-tunnel}` |

## Reachability from the Mac

Mac is on Starlink Wi-Fi; ms02 is on the Starlink LAN-out port. The router's
Wi-Fi↔LAN bridge filters arbitrary TCP (see
[starlink-wifi-lan-port-filter](../concepts/starlink-wifi-lan-port-filter.md)),
so **all dev traffic currently goes via the SSH tunnel**:

```bash
just dev-tunnel-up      # starts the multiplexed ssh tunnel
# http://localhost:10348/     → Tilt UI
# https://localhost:38839     → kube-apiserver (zero-rewrite kubeconfig)
# curl --socks5 127.0.0.1:1080 http://ms02:<port>/   → catch-all via SOCKS5
```

Retires the day a wired USB 2.5GbE adapter puts the Mac on the same switch as
ms02 (`ping -c 4 192.168.1.189` sub-ms = the signal to tear it down).

## Relevance to Sparks

- Build artifacts / Kind clusters for CI-like work land here, not on the Sparks.
- When we need a "client" to hit the vLLM API (`curl`, OpenAI Python SDK, eval harness),
  ms02 or the operator's laptop are the right places — keeping the Sparks doing only
  serving.
