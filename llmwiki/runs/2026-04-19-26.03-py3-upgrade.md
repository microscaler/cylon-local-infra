# First live upgrade — stack bounced from 26.01-py3 to 26.03-py3

## Context

After (a) shipping `ngc-image-sync` to keep NGC tags current and (b) shipping
`vllm-stack-autoupgrade` to automate the eventual cutover, the first actual
upgrade was performed today using the existing `roles/vllm_stacked_container`
role with a recreate flag — the same well-trusted path we'd use in any
operator-initiated cutover. The autoupgrade daemon was **deliberately left
disabled** for this first run; we wanted the explicit, observable path to
baseline the behaviour before trusting the automation to do it unattended.

## Pre-upgrade state

- Running leader container: `vllm-ngc-ray-head` on `nvcr.io/nvidia/vllm:26.01-py3`
- Running worker container: `vllm-ngc-ray-worker-nvidia2` on same `26.01-py3`
- No `vllm serve` process running inside head (had been killed during earlier
  HF prefetch work)
- HF cache: `TinyLlama/TinyLlama-1.1B-Chat-v1.0` only (Gemma-4, Qwen3 cleared
  during prior cleanup runs)
- Both Sparks already had `26.03-py3` pulled (via manual `docker save | ssh docker load` earlier, confirmed by `ngc-image-sync` as `ready`)

## Command

```bash
ansible-playbook playbooks/provision_sparks.yml \
  --skip-tags apt,hf_prefetch,vllm_autoupgrade,ngc_image_sync \
  -e vllm_stacked_container_recreate=true \
  -e vllm_default_model=TinyLlama/TinyLlama-1.1B-Chat-v1.0 \
  -e '{"vllm_api_server_extra_args":[]}'
```

Why per-run `-e` overrides instead of inventory edits:

- `vllm_default_model` in inventory still aspirationally points at
  `Qwen/Qwen3-30B-A3B-Instruct-2507` (a future target). That model is not yet
  in the HF cache.
- `vllm_api_server_extra_args: ["--max-model-len", "32768"]` in inventory is
  fine for Qwen3-30B but would crash TinyLlama (native max = 2048).
- Per-run `-e` overrides this single cutover to what's actually cached, no
  inventory churn, no footgun for later runs.

## Observed sequence

| Step | Outcome |
|---|---|
| Stop & rm old worker (nvidia2, 26.01) | ok |
| Stop & rm old head (nvidia1, 26.01) | ok |
| Pull `26.03-py3` | already local, no-op |
| Render `head.env` / `worker-nvidia2.env` | no change (NCCL env unchanged) |
| Start new head on `26.03-py3` | Ray head up, GCS on :6379 |
| Wait for GCS | ok |
| Start new worker on `26.03-py3` | joined Ray cluster |
| Pause 15 s for worker join | ok |
| Check for existing `vllm serve` inside head | none running (expected) |
| `docker exec -d head 'vllm serve TinyLlama … --tp 2 --host 0.0.0.0 --port 8000 --distributed-executor-backend ray'` | launched |

Total ansible run time: **~58 s**. vLLM engine + CUDA graph capture in the
new image: **~45 s** after that.

## Smoke test

```
curl http://nvidia1:8000/v1/models
  → 200 OK; served model: TinyLlama/TinyLlama-1.1B-Chat-v1.0, max_model_len=2048

curl http://nvidia1:8000/v1/chat/completions \
  -d '{"model":"TinyLlama/TinyLlama-1.1B-Chat-v1.0", …, "content":"Reply with the single word UPGRADED and nothing else."}'
  → {"content":"UPGRADED", "finish_reason":"stop"}

curl http://nvidia1:8000/metrics | grep -E '^vllm:(num_requests|request_success_total)'
  → vllm:num_requests_running{engine="0",model_name="…"} 0.0
  → vllm:num_requests_waiting{engine="0",model_name="…"} 0.0
  → vllm:request_success_total{engine="0",finished_reason="stop",model_name="…"} 1.0
  → …
```

The `/metrics` endpoint is present and exposes the counters
[vllm-stack-autoupgrade-service](../entities/vllm-stack-autoupgrade-service.md)
relies on for the quiet-window gate — confirmed as a byproduct of the smoke
test.

## Post-upgrade state across all three daemons

```
hf-prefetch:       ["TinyLlama/TinyLlama-1.1B-Chat-v1.0"]
ngc-image-sync:    [{"nvcr.io/nvidia/vllm:26.02-py3":"ready"},
                    {"nvcr.io/nvidia/vllm:26.03-py3":"ready"}]
autoupgrade:       {"current":"nvcr.io/nvidia/vllm:26.03-py3",
                    "status":"idle",
                    "enabled":false}
```

After `systemctl restart vllm-stack-autoupgrade` to force an immediate poll,
the autoupgrade daemon's `current_image` correctly re-read as 26.03-py3. With
nothing newer in `ngc-image-sync`'s `ready` set, the daemon transitioned to
`idle` — exactly the behaviour we want until a newer tag lands.
